> **작성일**: 2026-06-13  
> **대상**: 초등 4학년 / 대항해시대2 참고 / Phaser 3 + TypeScript

## 1. Context

초등학교 4학년 대상으로 세계 지리를 자연스럽게 학습하는 HTML5 브라우저 게임.  
대항해시대2의 핵심 재미인 "항구 도착 → 정보 수집 → 세계 발견" 루프를 교육적으로 재해석한다.  
6명의 전문 AI 에이전트가 director의 지휘 아래 자율적으로 개발하며, 사용자 개입은 최소화한다.

### 2-3. Director 작동 원칙
1. Task를 생성하여 각 에이전트에 위임
2. 완료 후 qa 에이전트 자동 호출 → pass/fail 판정
3. fail이면 해당 에이전트에 피드백 + 재작업
4. **사용자 호출 기준**: 게임의 핵심 방향성 변경, 예산/범위 결정, 에이전트 간 합의 불가 시
5. 그 외는 에이전트들끼리 상의하여 결정

## 3. Harness Engineering 워크플로우

director 에이전트가 실행하는 개발 파이프라인 (Workflow tool 활용):

```
Phase 1 — RESEARCH (병렬)
  ├── designer: 게임 시스템 설계 (대항해시대2 참고 → game-design-doc.md)
  └── writer:   콘텐츠 리서치 (30개 도시 × 특산품/사실 → cities-data.json)

Phase 2 — DESIGN REVIEW (병렬)
  ├── art:    스프라이트 설계 + 스타일 가이드 작성
  └── client: 기술 아키텍처 설계 (파일 구조, 씬 설계)

Phase 3 — IMPLEMENTATION (순차)
  client: Phase 2 산출물 기반으로 게임 코드 개발
    ├── Milestone A: 세계지도 + 캐릭터 선택
    ├── Milestone B: 항구 시스템 + 교육 카드
    └── Milestone C: 발견 기록장 + 칭호 시스템

Phase 4 — QA GATE (루프)
  qa: 각 Milestone 완료 시 테스트
    ├── pass → 다음 Milestone
    └── fail → client/designer에 버그 리포트 → 수정 후 재테스트

Phase 5 — POLISH
  art + writer + client: 피드백 반영, 효과음, 최종 콘텐츠
```

**Cross-agent 규칙**:
- 각 에이전트는 자신의 전담 디렉토리에만 쓰기
- 다른 에이전트 산출물은 읽기만 가능
- 충돌 시 director가 중재

## 4. 대항해시대2 참고 설계 (리서치 반영)

### 4-1. 핵심 루프 (우리 게임에 맞게 단순화)
```
항구 출발 → 세계지도 항해(방향키) → 항구 발견
  → 교육 카드 팝업(도시명/특산품/사실) → 발견 도장 획득
  → 기록장에 추가 → 다음 항구 탐색
```

원작의 복잡한 무역 경제 시스템은 제거, **발견과 학습**에 집중

### 4-2. 세계 지도 구조
원작의 13개 해역 구조 참고 → **6개 대륙 권역**으로 단순화:
- 유럽 / 아프리카 / 아시아 / 아메리카 / 오세아니아 / 극지방(보너스)

### 4-3. 항구 도시 (50개, 1차)
원작 100+개 → 초등 4학년 수준에 맞춰 **50개 주요 항구**로 선별

**아시아 (12개)**: 서울, 도쿄, 베이징, 상하이, 홍콩, 싱가포르, 방콕, 뭄바이, 콜카타, 자카르타, 하노이, 카라치  
**유럽 (15개)**: 리스본, 마드리드, 파리, 런던, 암스테르담, 베를린, 로마, 베네치아, 아테네, 이스탄불, 바르셀로나, 오슬로, 스톡홀름, 코펜하겐, 모스크바  
**아프리카 (8개)**: 카이로, 카사블랑카, 라고스, 케이프타운, 잔지바르, 나이로비, 알렉산드리아, 다카르  
**아메리카 (10개)**: 뉴욕, 워싱턴, LA, 시카고, 리우데자네이루, 부에노스아이레스, 아바나, 멕시코시티, 리마, 산티아고  
**오세아니아 (3개)**: 시드니, 멜버른, 오클랜드  
**중동 (2개)**: 두바이, 리야드

### 4-4. 특산품 시스템 (원작 46종 → 교육 최적화)
도시당 **3개 특산품** + **1개 랜드마크** + **1개 재미있는 사실**  
- 특산품 아이콘: 픽셀아트 아이콘 (64종 기본 세트)
- 언어: 한국어 주 + 영어 병기 (예: "김치 / Kimchi")

### 4-5. 발견 시스템 (원작 Discovery 참고)
- 새 항구 발견 시: 팡파레 + 발견 카드 애니메이션
- 발견 기록장(Logbook): 방문 도시 스탬프 + 세계지도 채우기
- 칭호 진행:
  - 0-5개: 견습 선원 (Apprentice Sailor)
  - 6-15개: 탐험가 (Explorer)
  - 16-30개: 항해사 (Navigator)
  - 31-50개: 세계 일주 달인 (World Voyager)

### 4-6. 캐릭터 시스템 (원작 6명 → 2명으로 단순화)
- **남자 캐릭터**: 항해사 준 (Jun the Navigator)
- **여자 캐릭터**: 탐험가 아라 (Ara the Explorer)
- 선택한 캐릭터가 말풍선으로 도시 소개
- 배 위 스프라이트 + 항구 NPC 대화 시 등장

## 7. 단계별 개발 원칙

> **핵심 원칙**: 각 단계 시작 전 → director + 관련 에이전트가 함께 세부 계획 수립 → 기능 구현 → 기능 테스트 → 전체 회귀 테스트 → 다음 단계

### 개발 흐름
```
[단계 N 시작]
  1. director가 해당 단계 세부 plan 작성 (designer/client와 함께)
  2. client 기능 구현
  3. qa 기능 테스트 (단위)
     ├── fail → client 수정 → 재테스트
     └── pass → 다음으로
  4. qa 전체 회귀 테스트 (이전 단계 포함)
     ├── fail → 원인 에이전트에 리포트 → 수정 → 재테스트
     └── pass → Git 커밋 → 다음 단계
```

### 확정된 설계 방향 (사용자 결정)
| 항목 | 결정 |
|------|------|
| 상점/거래 시스템 | **없음** — 도시 도착 후 특산품 확인 + 발견 도장만 |
| 도시 분포 | **균등 배분** — 대륙별 고르게 |
| 저장 방식 | **localStorage** |
| 배포 | 고려 안 함. **GitHub으로 형상관리** (단계별 커밋) |

### 단계 목록 (세부 계획은 각 단계 시작 전 수립)

**Step 1 — 프로젝트 기반**
- Vite + Phaser 3 + TypeScript 세팅
- Git 초기화 + GitHub 연결
- 빈 씬 3개 (Title, WorldMap, Port) 골격

**Step 2 — 캐릭터 선택**
- TitleScene + CharacterSelectScene
- 남자/여자 선택 → 저장 → WorldMap으로 전환

**Step 3 — 세계지도 + 배 이동**
- 세계지도 렌더링 (픽셀아트 스타일)
- 방향키/WASD 배 이동
- 지도 경계 처리

**Step 4 — 항구 시스템**
- 항구 50개 좌표 데이터
- 항구 근접 시 "닻" 아이콘 표시
- 항구 진입 판정

**Step 5 — 교육 카드 (핵심)**
- PortScene 카드 UI
- 도시명(한/영), 특산품 3개, 랜드마크, 재미있는 사실
- 발견 도장 + localStorage 저장

**Step 6 — 발견 기록장**
- LogbookScene
- 방문 도시 목록 + 세계지도 채우기 시각화

**Step 7 — 칭호 + 완성도**
- 칭호 4단계 시스템
- 픽셀아트 에셋 최종 적용
- 효과음

각 Step 완료 시: **Git 커밋 + GitHub push** (형상관리)

## 10. 사용자 개입 기준 (director 판단)

director가 자율 결정 가능:
- 기술 구현 방법 선택
- 에이전트 간 역할 분배
- 버그 수정 방향
- 단계 내 세부 설계

**사용자 호출 필요 (꼭 확인 후 진행):**
- 게임의 핵심 교육 목표/방향이 바뀌는 결정
- 예상 불가한 기술적 장벽으로 대안 구조 필요 시
- 단계 테스트에서 근본적 설계 문제 발견 시
